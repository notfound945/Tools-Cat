## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Tight inline gaps inside status rows and between spinner/icon and text |
| sm | 8px | Radio icon to label spacing; compact vertical gaps inside one control group |
| md | 16px | Default row spacing in the WOL window; default content padding |
| lg | 24px | Separation between form body and trailing action row |
| xl | 32px | Reserved for larger window-section gaps if Phase 1 needs a secondary status block |
| 2xl | 48px | Reserved for future larger native sheets; not for the compact Phase 1 window |
| 3xl | 64px | Not used in this phase; keep for later management surfaces |

Exceptions: native `NSMenuItem` row height, status bar icon sizing, and standard AppKit control insets remain system-managed; keep the WOL window compact at roughly 420-460px wide rather than inventing custom page gutters. Source: current codebase plus default.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 13px | 400 | 1.5 |
| Label | 13px | 600 | 1.3 |
| Heading | 17px | 600 | 1.2 |
| Display | 22px | 600 | 1.2 |

Use only these three sizes in Phase 1. Menu labels and field copy stay at Body or Label size; the WOL window title uses Heading; Display is reserved for no more than one future emphasis line and should not appear in the compact menu. Default chosen to match native macOS utility density.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `NSColor.windowBackgroundColor` (approx `#F5F5F5` light / `#1E1E1E` dark) | WOL window background and the dominant surface of the phase |
| Secondary (30%) | `NSColor.controlBackgroundColor` (approx `#EBEBEB` light / `#2C2C2E` dark) | Text field fill, inset status region, passive grouped surfaces |
| Accent (10%) | `NSColor.controlAccentColor` (default macOS blue, approx `#007AFF`) | Enabled primary CTA fill, selected radio glyph, active progress indicator only |
| Destructive | `NSColor.systemRed` (`#FF3B30`) | Destructive actions only; not expected to appear in Phase 1 |

Accent reserved for: the enabled `发送唤醒包` button, the selected input-mode radio symbol, and the in-flight progress indicator. Never use accent for validation copy, failure copy, success copy, passive menu labels, or keep-awake steady-state checkmarks. Validation/failure states should use `NSColor.systemOrange`; confirmed local-send success should use `NSColor.systemGreen`; idle helper text should use `NSColor.secondaryLabelColor`. Source: default, constrained by 01-CONTEXT.md truthful-state requirements.

## Interaction Contract

- The primary visual anchor in the WOL window is the MAC entry plus `发送唤醒包` action row. First-read order should be: title, input mode and MAC entry control, primary action, then the status area as secondary feedback beneath the form. This phase should feel like a compact native utility dialog, not a dashboard.
- Manual MAC validation runs on every keystroke in custom-entry mode. Accept free typing, but do not auto-insert separators or rewrite case for the user while typing. Source: D-01 through D-05.
- The send CTA stays disabled until the manual MAC is fully valid in `AA:BB:CC:DD:EE:FF` format and no send is currently in progress. Source: D-02, D-03.
- The status area inside the WOL window is the only result surface in this phase. It supports five states: idle, validation error, sending, local-send success, and local-send failure. Source: D-06 through D-09.
- Starting a new send clears the previous result immediately. While sending, keep the current window open state-independent from the network task and show progress inline next to or inside the primary CTA. Source: D-08, D-16.
- Success copy must mean only that the packet was sent from this Mac. Never imply the target device is awake, reachable, or verified. Source: D-06.
- Failure copy must stay human-readable, explain the local failure, and suggest the next action when possible. Do not expose raw Swift error text or errno strings. Source: D-07.
- Reopening the WOL window preserves unfinished input but resets the visible result area to idle. If a send finished while the window was closed, show only that final result after reopen. Source: D-14 through D-17.
- Keep-awake presentation has separate confirmed and pending states: `off`, `turning on`, `on`, `turning off`. The menu item label carries the pending copy; the checkmark and menu bar icon change only after confirmed success. Source: D-10 through D-13.
- If a keep-awake toggle fails, keep the previous confirmed icon/checkmark state and surface one clear failure message in the menu context or adjacent native feedback surface; do not silently snap to the requested state. Source: D-13.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | `发送唤醒包` |
| Empty state heading | `尚未发送唤醒请求` |
| Empty state body | `输入有效的 MAC 地址后，再发送唤醒包。重新打开窗口时，这里回到空状态。` |
| Error state | `先说明问题，再给下一步。验证错误直接指出格式问题；发送失败明确说明“未能从这台 Mac 发出唤醒包”，并提示检查局域网连接后重试；保持常亮失败说明状态未改变。` |
| Destructive confirmation | `无`: 本阶段不新增删除、覆盖或退出前确认；关闭窗口不弹确认，未完成输入在会话内保留 |

Additional copy rules:

- Validation copy is specific by error type. Use messages equivalent to `请填写 MAC 地址`, `请输入冒号分隔格式，例如 AA:BB:CC:DD:EE:FF`, and `MAC 地址必须是 6 组两位十六进制字符`.
- Success copy uses local-send language only, for example `已从这台 Mac 发出唤醒包`.
- Sending copy should stay neutral, for example `正在发送唤醒包...`.
- Keep-awake pending labels should read as inline menu states, for example `保持屏幕常亮（正在开启...）` and `保持屏幕常亮（正在关闭...）`.

Source: 01-CONTEXT.md decisions plus default Chinese runtime-string convention from codebase and CLAUDE.md.
