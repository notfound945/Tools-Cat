## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Name-to-MAC stack gap, inline icon spacing |
| sm | 8px | Field label to helper/error text, compact button groups |
| md | 16px | Default section padding, list row padding, control gaps |
| lg | 24px | Header/footer padding and list-to-actions separation |
| xl | 32px | Empty-state vertical spacing and form section breathing room |
| 2xl | 48px | Major internal mode transition spacing, top-to-bottom empty-state block |
| 3xl | 64px | Reserved for future large surfaces; not used as a default in this compact window |

Exceptions: `40px` minimum height for primary text fields and action buttons, `56px` minimum device-row height, `72px` minimum note field height. (default, phase-specific)

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.5 |
| Label | 12px | 600 | 1.3 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.15 |

Use only weights `400` and `600`. Device names and MAC addresses both use the Body size so they scan as equal-priority identifiers; device names use weight `600`, MAC addresses use weight `400` with monospaced digits.

## Color

Use macOS semantic system colors in implementation; the values below are light-mode reference targets for the contract, not hardcoded palette-only requirements.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | #ECECEC | Window background, outer surface, empty-state canvas |
| Secondary (30%) | #FFFFFF | List rows, form group surfaces, inset content blocks |
| Accent (10%) | #0A84FF | Primary save button, active reorder toggle, focused field ring, selected row outline |
| Destructive | #FF453A | Delete button, delete confirmation, hard-stop persistence errors |

Accent reserved for: primary save CTA, active reorder-state affordance, current input focus, and selected list row outline only. Do not use accent for secondary buttons, passive list rows, helper text, or empty-state art.

## Native Surface Contract

- Management lives in a dedicated titled, closable native window and remains independent from the WOL send window. Both windows may stay open at the same time. (source: `02-CONTEXT.md` D-01 to D-03)
- Default window title: `设备库`. Initial size: `520x420`. Keep the window compact and utility-like; do not introduce a sidebar, split view, preferences toolbar, or multi-pane inspector. (source: `02-CONTEXT.md` D-02, default dimensions)
- Add a dedicated menu item labeled `管理设备…` directly below `发送 WOL …` and above the first separator so management stays discoverable without crowding the compact menu. (source: `02-CONTEXT.md` D-04, default placement)
- The management window uses exactly two internal modes: `list` as the default state and `form` as the secondary state for both create and edit. Switching modes replaces the body content inside the same window; do not open a second editor window. (source: `02-CONTEXT.md` D-08 to D-10)

## Layout Contract

- In the default `list` mode, the device list body is the primary visual anchor of the window. Header actions remain secondary utility controls and must not visually overpower the saved-device rows.
- List mode header: left-aligned title `已保存设备`, right-aligned actions `重新排序` and `添加设备`.
- Footer actions are right-aligned. Secondary actions use bordered controls; the primary save action uses bordered prominent styling to match the existing WOL window.
- Keep outer window padding at `24px`, inner content spacing at `16px`, and action-group spacing at `8px`.
- Empty state is vertically centered inside the content area with a secondary-tone SF Symbol, a Heading, then Body copy, then a bordered prominent `添加设备` button.

## Device Row Contract

- Each row is a balanced stacked layout with three possible lines: device name, MAC address, optional note preview.
- Device name and MAC address are both visually primary. Render both in `14px` body size; device name uses weight `600`, MAC address uses weight `400` and monospaced digits. Both use primary text color.
- Note preview is optional, one line maximum, `12px` secondary text, truncated with tail ellipsis.
- Use `16px` internal row padding and `4px` between the name and MAC lines.
- Passive list mode shows edit access but no drag handles. Reorder affordances appear only while reorder mode is active. (source: `02-CONTEXT.md` D-11)

## Form Contract

- Field order is fixed: `名称`, `MAC 地址`, `备注（可选）`.
- `名称` is a single-line text field and trims leading/trailing whitespace before save.
- `MAC 地址` is a single-line text field that reuses `ManualMACValidator`; inline guidance and blocking validation must match the manual WOL send rules from Phase 1.
- Normalize MAC addresses to uppercase colon-separated format on successful validation or save.
- `备注` is a multiline field capped visually at three lines in the editor and one line in list preview.
- Add and edit use the same layout, field order, validation behavior, and footer actions. The only copy change is the mode title: `添加设备` or `编辑设备`. (source: `02-CONTEXT.md` D-09)

## Interaction Contract

- Opening `管理设备…` brings the management window to the front and activates the app, mirroring the current WOL window behavior.
- `添加设备` enters form mode with empty fields. `编辑` enters the same form with existing values prefilled.
- `取消` always returns from form mode to list mode without mutating persisted data.
- `保存设备` stays disabled until the trimmed name is non-empty and `ManualMACValidator` reports a valid MAC.
- Validation errors appear inline below the relevant field before save is attempted. Use the existing truthful-validation pattern: block invalid edits; do not allow optimistic saves. (source: `RELY-01`, `02-RESEARCH.md`)
- Reorder is explicit. Entering reorder mode reveals drag affordances and changes the header action label from `重新排序` to `完成排序`.
- While reorder mode is active, hide or disable edit and delete actions to prevent accidental mixed-mode interactions in the compact window.
- Persist order immediately after a completed drag move and preserve that order on relaunch. (source: `DEVS-04`, `02-CONTEXT.md` D-13)
- Delete uses a standard confirmation alert attached to the management window, with `取消` as the default action and `删除设备` as the destructive action. Never delete inline without confirmation. (source: `02-CONTEXT.md` D-12)

## State Contract

- Empty state: no devices yet; show the empty-state block and a single primary `添加设备` action.
- Populated list state: show rows in persisted order, newest additions appended unless the user later reorders them.
- Form validation state: keep the form visible, highlight the invalid field, show inline error text, and do not dismiss the editor.
- Save failure state: keep the form open and show the error copy near the footer so the user can retry without losing input.
- Delete result state: after a confirmed delete, remove the row immediately from the list and fall back to empty state if the library becomes empty.

## Component Inventory

- Menu item: `管理设备…`
- Management window shell: titled, closable AppKit window hosting a SwiftUI view
- Header action bar: title plus `重新排序` and `添加设备`
- Device list row: stacked name/MAC/note preview plus trailing row actions
- Empty state block: SF Symbol, heading, body, primary action
- Shared device form: name field, MAC field, note field, inline validation, cancel/save footer
- Delete confirmation alert: cancel + destructive delete

## Out of Scope Guardrails

- Do not add deep inline editing inside the menu bar menu.
- Do not introduce recent devices, wake shortcuts, last-used device behavior, import/export, LAN discovery, or advanced networking fields in this phase.
- Do not turn this surface into a general settings window, split-pane device manager, or database-backed inventory tool.
