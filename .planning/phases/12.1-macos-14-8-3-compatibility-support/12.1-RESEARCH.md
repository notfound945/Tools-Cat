# Phase 12.1: macOS 14.8.3 Compatibility Support - Research

**Researched:** 2026-04-15
**Domain:** Deployment-target compatibility for a native macOS menu bar app
**Confidence:** HIGH

## Summary

The current incompatibility is configuration truth, not a broad runtime rewrite. The repo was excluding macOS 14.8.3 because `Tools Cat.xcodeproj/project.pbxproj` set `MACOSX_DEPLOYMENT_TARGET = 15.6` for the project and test targets. After lowering that baseline to `14.0`, `xcodebuild -showBuildSettings` reports `MACOSX_DEPLOYMENT_TARGET = 14.0`, and a local unsigned build succeeds.

The code scan did not reveal obvious macOS 15-only APIs in the app sources. The app is built around AppKit, SwiftUI, Combine, and `ObservableObject`, all of which are already available on macOS 14. That means Phase 12.1 should stay narrow: lock the deployment target fix, update any stale documentation that still claims a 15.6 minimum, and record a verification boundary that separates "builds against a 14.x target" from "actually launches on Sonoma 14.8.3".

## Existing Compatibility Findings

### Confirmed Blocking Point

- `Tools Cat.xcodeproj/project.pbxproj` previously set `MACOSX_DEPLOYMENT_TARGET = 15.6` in four places.
- That alone is enough to prevent the app from being distributed as a Sonoma-compatible build.

### Code Scan Result

The app source scan did not find obvious usage of common newer-only SwiftUI APIs such as:

- `MenuBarExtra`
- `NavigationSplitView`
- `Table`
- `symbolEffect`
- `persistentSystemOverlays`
- `sensoryFeedback`
- `@Observable`

Current observable/state patterns use `ObservableObject`, which is compatible with macOS 14.

### Documentation Drift

- `CLAUDE.md` still says: `macOS 15.6 minimum deployment target`.
- That document is now false once the project target is lowered, so compatibility work is incomplete unless the docs are corrected.

## Recommended Phase Boundary

This phase should only cover:

1. Align the deployment target baseline to a Sonoma-compatible value.
2. Remove stale project guidance that still claims 15.6 is the minimum supported version.
3. Record verification evidence that the app builds against macOS 14 and define the one manual-only boundary for real 14.8.3 launch validation.

Do not use this phase to reopen:

- notarization/signing
- broader distribution hardening
- UI redesign
- duration-management scope

## Technical Direction

### Deployment Target

Use `MACOSX_DEPLOYMENT_TARGET = 14.0` consistently across:

- project build configurations
- test target build configurations

`14.0` is the correct baseline if the app must run on `14.8.3`, because the patch release is within the same major/minor deployment family.

### Documentation Truth

Update repo guidance that currently bakes in `15.6` as the minimum. The most obvious place is:

- `CLAUDE.md`

If any release or support docs mention Sonoma incompatibility or 15.x-only support, align those too. The phase should not leave the code and docs disagreeing.

### Verification Boundary

There are two different truths to capture:

1. **Automated truth:** the app now builds with `MACOSX_DEPLOYMENT_TARGET = 14.0`
2. **Manual truth:** a real macOS 14.8.3 machine can launch the built app

The first can be automated locally with:

- `xcodebuild -showBuildSettings`
- unsigned `xcodebuild build`

The second requires an explicit manual boundary because this machine is not guaranteed to be on macOS 14.8.3.

## Recommended Plan Shape

One plan is enough.

Suggested scope:

- normalize deployment target in the Xcode project
- update stale compatibility docs
- add one verification artifact or validation contract for the 14.8.3 boundary

## Validation Architecture

### Automated Checks

- `xcodebuild -project "Tools Cat.xcodeproj" -scheme "Tools Cat" -showBuildSettings | rg "MACOSX_DEPLOYMENT_TARGET = 14.0"`
- `xcodebuild -project "Tools Cat.xcodeproj" -scheme "Tools Cat" -destination 'platform=macOS' CODE_SIGNING_ALLOWED=NO build`

These prove the shipped build baseline is no longer pinned to 15.6.

### Manual Boundary

One explicit manual compatibility smoke should remain:

- on a real macOS 14.8.3 machine, launch the built app and confirm the menu bar item appears and the app does not immediately exit because of OS-version incompatibility

That manual boundary is acceptable because OS-version runtime confirmation cannot be proven from this machine alone.

## Pitfalls

### Pitfall 1: Fixing Only the App Target

If tests or other target configs stay on 15.6, the project remains internally inconsistent and future tooling/drift can reintroduce confusion.

### Pitfall 2: Leaving `CLAUDE.md` Wrong

Planning and codebase guidance that still says "minimum 15.6" will cause future agents to accidentally revert or mis-explain the compatibility baseline.

### Pitfall 3: Claiming Full Sonoma Runtime Support From Build-Only Evidence

Lowering the deployment target and getting a successful local build is strong evidence, but it is not the same as a real launch on 14.8.3. The phase should keep that distinction explicit.

## Decision

Phase 12.1 should be a narrow compatibility-truth phase: align deployment target to macOS 14, update stale docs that still claim 15.6, and lock the evidence chain with automated build checks plus one explicit manual 14.8.3 launch boundary.
