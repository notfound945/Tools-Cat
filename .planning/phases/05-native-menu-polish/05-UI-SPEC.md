## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline metadata spacing `[default]` |
| sm | 8px | Label-to-field spacing, trailing text-button gaps `[default]` |
| md | 16px | Default stack spacing in compact utility views `[source: existing WOLView.swift and DeviceLibraryView.swift]` |
| lg | 24px | Window padding and top action breathing room `[source: existing DeviceLibraryView.swift + default]` |
| xl | 32px | Empty-state block spacing and title-to-section breaks `[default]` |
| 2xl | 48px | Major empty-state vertical separation `[default]` |
| 3xl | 64px | Reserve only for full-height empty-state centering; do not use in the root menu `[default]` |

Exceptions: Native `NSMenuItem` row heights remain system-controlled; compactness comes from showing fewer rows, not adding custom row padding. Keep the existing compact window shells as the baseline: WOL window at 460pt width and device-library window at 520x420 minimum `[source: WOLWindow.swift, DeviceLibraryWindow.swift, 05-CONTEXT.md D-07 D-10]`.

Source notes: The 8-point scale is standardized here as the Phase 5 contract. Existing SwiftUI views already cluster around 16pt and 24pt spacing; this contract formalizes the rest.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `windowBackgroundColor` (reference `#F5F5F7`) | WOL window and device-library window backgrounds `[default: native macOS surface]` |
| Secondary (30%) | `controlBackgroundColor` / `separatorColor` (reference `#EBEBEF`) | Text fields, picker surfaces, list dividers, quiet grouping boundaries `[default: native macOS controls]` |
| Accent (10%) | `controlAccentColor` (reference `#0A84FF`) | `发送唤醒包`, `添加设备`, selected mode indicator, keyboard focus ring only `[source: existing prominent actions + default system accent]` |
| Destructive | `systemRed` (`#FF3B30`) | `删除` action, destructive alert button only `[source: existing delete role]` |

Accent reserved for: primary CTA buttons, the selected WOL input-mode control, and focus indication. Do not use accent color for status copy, passive menu rows, section labels, or decorative chrome.

Source notes: Implement with semantic system colors, not hard-coded fills. Reference hex values exist only to make the checker contract concrete.

## Root Menu Contract

| Section | Required Items | Visibility Rules |
|---------|----------------|------------------|
| Keep-awake | `无限常亮`, `15 分钟`, `30 分钟`, `1 小时`, `2 小时`, `关闭常亮`, optional status row | Always keep these action rows at the top of the root menu. Show the keep-awake status row only when the state is pending, actively on, timed with countdown, or carrying a truthful message/failure `[source: 05-CONTEXT.md D-01 D-04 D-05 D-06, KeepAwakePresentation.swift]` |
| Wake | Up to 3 recent-device rows, `所有设备`, `发送 WOL …`, optional wake status row | Recent rows appear only when recents exist. `所有设备` appears only when the library is non-empty. Show exactly one wake status row, as the last row in the wake section, only while sending or when a truthful last-completed result exists `[source: 05-CONTEXT.md D-03 D-04 D-05, StatusBarController.swift]` |
| Management | `管理设备…`, `退出 Swiss Knife` | Always place these rows last. Do not add explanatory rows or extra actions in this phase `[source: 05-CONTEXT.md D-01 D-02]` |

Rules:
- Use one native separator between keep-awake and wake, and one native separator between wake and management. Do not use text headers inside the menu `[source: 05-CONTEXT.md D-02]`.
- Do not add icons, badges, or attributed rich text to menu rows. Use native checkmarks and the existing status-bar symbol only `[source: 05-CONTEXT.md D-09, codebase]`.
- Disable all keep-awake action rows while a keep-awake state change is pending `[source: KeepAwakePresentation.swift]`.
- Disable all wake-trigger rows while a wake send is in progress to preserve the no-duplicate-send guarantee `[source: REQUIREMENTS.md RELY-04, StatusBarController.swift]`.
- Idle menu state must collapse both status rows when they carry no meaningful information `[source: 05-CONTEXT.md D-04 D-05]`.

## WOL Window Contract

| Area | Contract |
|------|----------|
| Shell | Keep a compact titled utility window. Retain the current single-column structure and 460pt width baseline; any height increase must stay minimal and serve hierarchy, not add chrome `[source: 05-CONTEXT.md D-07 D-08, WOLWindow.swift]` |
| Title block | One heading line only: `唤醒局域网设备 (WOL)`. No subtitle by default. Use heading typography and leave 16pt before the mode selector `[source: existing WOLView.swift + 05-CONTEXT.md D-09]` |
| Mode selection | Keep the current two-option horizontal mode choice. Accent is limited to the selected indicator; labels stay primary text. No new tabs, segmented control migrations, or extra modes in Phase 5 `[source: 05-CONTEXT.md D-09, WOLView.swift]` |
| Input area | Show exactly one input control at a time: manual MAC text field or saved-device picker. Inputs use rounded native controls and align to the full content width `[source: WOLView.swift]` |
| Status area | Reserve a single 1-2 line text block directly above the action row. Hide it when there is no validation, sending, success, or failure message. Use secondary for in-progress, green for success, orange for validation/failure `[source: WOLView.swift, REQUIREMENTS.md RELY-02 RELY-03]` |
| Action row | Trailing-aligned `取消` secondary button and `发送唤醒包` prominent primary button with 16pt separation. Keep the primary button disabled when the session cannot send or is already sending `[source: WOLView.swift, REQUIREMENTS.md RELY-04]` |

Rules:
- Use 16pt spacing between major sections and 8pt inside a control group.
- Do not introduce cards, shadows, tinted panels, or dashboard-style framing.
- Preserve truthful status semantics from earlier phases; polish may shorten or hide idle feedback but must not replace exact failure reasons with generic optimism.

## Device-Library Window Contract

| Area | Contract |
|------|----------|
| Shell | Keep a compact titled management window with the saved-device list as the visual anchor. Retain the current 520x420 minimum shell unless a small increase is needed for cleaner spacing `[source: 05-CONTEXT.md D-10 D-12, DeviceLibraryWindow.swift]` |
| Top action row | Title left, then trailing actions: `重新排序`/`完成排序` as secondary, `添加设备` as prominent primary. Maintain one horizontal row with 8pt button spacing `[source: DeviceLibraryView.swift, DeviceLibraryManagementPresentation.swift]` |
| Empty state | Center one SF Symbol, one heading, one supporting paragraph, and one `添加设备` button. Use 32pt between icon and heading cluster, and keep copy to two lines max where possible `[source: 05-CONTEXT.md D-11, DeviceLibraryView.swift]` |
| List rows | Each populated row stacks device name, MAC address, and optional note. Name is the strongest line, MAC uses monospaced digits, note is caption-sized secondary text. Keep 16pt vertical row padding and use dividers only between rows `[source: DeviceLibraryView.swift]` |
| Row actions | In normal mode, show trailing text buttons `编辑` and `删除`. In reorder mode, replace those actions with the drag handle only. Do not mix drag affordance with edit/delete controls `[source: existing DeviceLibraryView.swift]` |
| Form mode | Keep labels above controls, validation copy directly under the affected field, and the save-error message above the action row. The form stays secondary to the list-first surface `[source: 05-CONTEXT.md D-12, DeviceLibraryView.swift]` |

Rules:
- Keep list mode visually lighter than form mode; the list is the primary surface for this phase.
- Do not add persistent sidebars, split panes, inspector panels, or inline row editing.
- Preserve the destructive delete confirmation as a modal alert using the existing destructive wording.

## Interaction Guardrails

- Keep the app small and scannable. Phase 5 may refine grouping and hierarchy, but it must not add new capabilities, onboarding banners, or persistent helper text `[source: 05-CONTEXT.md phase boundary]`.
- Menu polish must stay native: separators over headings, checkmarks over custom badges, and status rows only when they communicate real state `[source: 05-CONTEXT.md D-02 D-04 D-06]`.
- Window polish must come from spacing, typography, grouping, and restrained copy. Do not introduce custom illustrations, tinted hero panels, or non-system motion `[source: 05-CONTEXT.md D-09 D-11]`.
- Keep runtime strings in Chinese and code identifiers in English `[source: codebase conventions]`.
