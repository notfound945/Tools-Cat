# Requirements: Mac OS Swiss Knife

**Defined:** 2026-04-11
**Core Value:** From the menu bar, I can reliably wake the devices I care about and trust the app's status without editing code or fighting the UI.

## v1 Requirements

Requirements for the next milestone of this existing macOS menu bar utility. These focus on daily-use WOL management, truthful feedback, and a more native, maintainable user experience.

### Device Library

- [x] **DEVS-01**: User can save a Wake-on-LAN device locally with a display name and MAC address
- [x] **DEVS-02**: User can edit a saved device's name, MAC address, and note
- [x] **DEVS-03**: User can delete a saved device
- [x] **DEVS-04**: User can reorder saved devices to match personal usage preference
- [x] **DEVS-05**: User can add an optional short note to a saved device to distinguish similar machines

### Wake Flow

- [x] **WOL-01**: User can wake any saved device directly from the menu bar without editing source code
- [x] **WOL-02**: User can still send a Wake-on-LAN packet by manually entering a MAC address when needed
- [x] **WOL-03**: User sees the most recently used saved device preselected when reopening the Wake-on-LAN window

### Keep Awake

- [x] **AWAKE-01**: User can enable keep-awake in either indefinite mode or timed mode from native menu controls
- [x] **AWAKE-02**: User can choose from a small set of preset keep-awake durations for timed mode
- [x] **AWAKE-03**: User sees a live countdown while a timed keep-awake session is active
- [x] **AWAKE-04**: Keep-awake turns off automatically when the selected timed session expires

### Reliability

- [x] **RELY-01**: User sees validation errors before an invalid device can be saved
- [x] **RELY-02**: User sees validation errors before an invalid Wake-on-LAN send is attempted
- [x] **RELY-03**: User sees whether the app successfully sent the Wake-on-LAN packet locally or why the local send failed
- [x] **RELY-04**: User cannot accidentally trigger duplicate Wake-on-LAN sends while a send is already in progress
- [x] **RELY-05**: User sees keep-awake menu state only when the underlying display-sleep assertion state actually changed

### Native UX

- [x] **UX-01**: User sees wake actions, keep-awake state, status feedback, and management actions grouped in a compact native macOS menu structure
- [x] **UX-02**: User can open a dedicated native management surface to add, edit, delete, and reorder saved devices
- [x] **UX-03**: User sees the result of the most recent wake attempt without needing to reopen or reinterpret the UI
- [x] **UX-04**: User experiences a visually restrained, polished interface with clear hierarchy and status cues consistent with native macOS expectations

## v2 Requirements

Deferred to a later release after the daily-use milestone is stable.

### Convenience

- **CONV-01**: User can pin favorite devices separately from recents
- **CONV-02**: User can trigger Wake Last Device with a dedicated keyboard shortcut
- **CONV-03**: User can import or export their saved device list
- **CONV-04**: User can access a short recent-devices list for faster repeat wake actions

### Advanced Networking

- **NET-01**: User can configure advanced per-device networking fields such as custom broadcast targets or interfaces
- **NET-02**: User can run additional reachability or wake-verification checks beyond local packet-send status

### Shell Modernization

- **SHELL-01**: App migrates the current AppKit shell to a `MenuBarExtra`-first implementation if post-hardening maintenance evidence justifies it

## Out of Scope

Explicitly excluded from this milestone to keep the app small, native, and focused on personal daily use.

| Feature | Reason |
|---------|--------|
| Automatic LAN discovery or scanning | Adds network complexity and uncertain value for a small manually managed personal device list |
| Cloud sync or shared device libraries | Misaligned with local-first self-use scope and adds identity/conflict complexity |
| Batch wake orchestration, scripts, or automation rules | Turns a quick utility into an automation system and raises debugging cost |
| Deep inline editing inside the menu bar menu | Conflicts with a compact native menu and should live in a dedicated management surface |
| Public-distribution polish as the primary milestone | Daily-use reliability and usability are the current priority over packaging hardening |
