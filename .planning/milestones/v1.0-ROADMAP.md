# Roadmap: Mac OS Swiss Knife

## Overview

This milestone turns the existing menu bar utility into a dependable daily-use tool by fixing truth and ownership first, then layering in saved-device management, faster wake workflows, timed keep-awake, and restrained native polish without drifting into discovery, cloud, or automation scope.

## Phase Details

### Phase 1: Truthful Foundations
**Goal**: Users can trust wake and keep-awake feedback because service outcomes, validation, and lifecycle-owned state stay aligned with real local results.
**Depends on**: Nothing (first phase)
**Requirements**: WOL-02, RELY-02, RELY-03, RELY-05
**Success Criteria** (what must be TRUE):
  1. User can manually enter a MAC address and sees validation errors before an invalid wake attempt is sent.
  2. User sees whether a wake request was sent locally or why the local send failed.
  3. Keep-awake menu state changes only after the underlying display-sleep assertion actually changes.
**Plans**: 3/3 plans complete

Plans:
- [x] 01-01-PLAN.md — Create truth contracts and unit-test harnesses for validation, send feedback, session state, and keep-awake presentation
- [x] 01-02-PLAN.md — Rewire the WOL window around a retained session model with real-time validation and truthful local-send results
- [x] 01-03-PLAN.md — Rework keep-awake menu state so only confirmed assertion outcomes drive the label, checkmark, icon, and failure feedback

### Phase 2: Device Library Management
**Goal**: Users can manage a small local Wake-on-LAN device library in a dedicated native surface instead of editing source code.
**Depends on**: Phase 1
**Requirements**: DEVS-01, DEVS-02, DEVS-03, DEVS-04, DEVS-05, RELY-01, UX-02
**Success Criteria** (what must be TRUE):
  1. User can open a dedicated native management surface to add a saved device with a name, MAC address, and optional note.
  2. User can edit or delete a saved device, and invalid edits are blocked before they are saved.
  3. User can reorder saved devices and sees that order preserved when the app is reopened.
**Plans**: 5/5 plans complete
Plans:
- [x] 02-01-PLAN.md — Establish saved-device persistence, shared library state, and management-session contracts with XCTest coverage
- [x] 02-02-PLAN.md — Build the compact device-library window, list/form flow, and dedicated menu entry
- [x] 02-03-PLAN.md — Replace hardcoded WOL presets with the shared saved-device library and add a UI smoke seam
- [x] 02-04-PLAN.md — Make seeded manager rows queryable under XCUITest and turn the seeded manager launch path into a passing regression gate
- [x] 02-05-PLAN.md — Wire the live manager window and view to the exact-copy presentation contract and close the remaining Phase 2 verification gap
**UI hint**: yes

### Phase 3: Saved-Device Wake Flows
**Goal**: Users can wake saved devices quickly from the menu bar and reuse recent context without duplicate or ambiguous wake actions.
**Depends on**: Phase 2
**Requirements**: WOL-01, WOL-03, WOL-04, RELY-04, UX-03
**Success Criteria** (what must be TRUE):
  1. User can wake any saved device directly from the menu bar without editing source code.
  2. User sees the most recently used saved device preselected when reopening the Wake-on-LAN window.
  3. User can access a short recent-devices list for faster repeat wake actions.
  4. User cannot trigger duplicate wake sends while another send is in progress, and the last wake result remains visible afterward.
**Plans**: 3/3 plans complete
Plans:
- [x] 03-01-PLAN.md — Add shared persistence and session contracts for recent-device memory, last-used memory, and durable last-result state
- [x] 03-02-PLAN.md — Wire compact menu wake actions, recent-device access, and persistent status rows onto the shared session
- [x] 03-03-PLAN.md — Reopen the WOL window with last-used saved-device defaults while preserving unfinished manual drafts
**UI hint**: yes

### Phase 4: Timed Keep-Awake
**Goal**: Users can control keep-awake as either an indefinite or timed session with explicit duration choices and countdown feedback.
**Depends on**: Phase 1
**Requirements**: AWAKE-01, AWAKE-02, AWAKE-03, AWAKE-04
**Success Criteria** (what must be TRUE):
  1. User can start keep-awake in indefinite mode or timed mode from native menu controls.
  2. User can choose from a small set of preset timed durations.
  3. User sees a live countdown while a timed keep-awake session is active.
  4. Keep-awake turns off automatically when the selected timed session expires.
**Plans**: 3/3 plans complete
Plans:
- [x] 04-01-PLAN.md — Build the timed keep-awake session model, fixed presets, countdown scheduler seam, and exact presentation contract
- [x] 04-02-PLAN.md — Rebuild the root keep-awake menu group around the shared session with fixed rows, stable titles, and controller tests
- [x] 04-03-PLAN.md — Run the live timed keep-awake menu smoke and confirm replacement, countdown, manual stop, and expiry behavior
**UI hint**: yes

### Phase 5: Native Menu Polish
**Goal**: Users experience a compact, clearly grouped, and visually restrained native macOS utility across the menu bar and management surfaces.
**Depends on**: Phase 3, Phase 4
**Requirements**: UX-01, UX-04

  2. User experiences restrained, polished wake and management surfaces with clear hierarchy and status cues consistent with native macOS expectations.
  3. The app stays small and scannable while still exposing saved devices, recent wake actions, and timed-session feedback.
**Plans**: 3/3 plans complete

- [x] 05-03-PLAN.md — Add direct UI-smoke coverage for both polished windows and finish with a blocking visual approval gate
